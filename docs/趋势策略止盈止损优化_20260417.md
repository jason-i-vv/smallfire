# 趋势策略止盈止损优化

> 日期：2026-04-17

## 背景

趋势策略(TrendStrategy)在生成信号时不计算止盈止损价格，完全依赖回测引擎的全局固定百分比（默认 2% 止损 / 5% 止盈）。

在 ETH 1H 趋势策略回测中发现：
- 止损 2% 时：频繁被正常回调扫损，整体亏损
- 止损 3% 时：盈利

根本原因：趋势策略捕捉的是 EMA 回踩后的趋势延续，需要给价格足够的回调空间。固定百分比止损在不同波动环境下表现不一致。

## 优化方案

### 基于 ATR 的动态止盈止损

使用 ATR（Average True Range，平均真实波幅）来动态计算止损距离：
- **止损距离** = ATR × 倍数（默认 2.0）
- **止盈距离** = 止损距离 × 盈亏比（默认 2.0）
- 高波动时止损更宽，低波动时止损更紧，自适应市场节奏

### 最低保护机制

- 止损距离不低于入场价的 1%（防止 ATR 极小时止损过近）
- 止盈距离不低于入场价的 2%（确保最小盈利空间）

### 信号级止盈止损

策略在生成信号时计算并写入 `Signal.StopLossPrice` 和 `Signal.TargetPrice`，回测引擎优先使用策略计算的绝对价格，而非全局百分比。

## 配置参数

```yaml
trend:
  enabled: true
  ema_periods: [30, 60, 90]
  check_interval: 60
  atr_period: 14              # ATR 计算周期
  atr_multiplier: 2.0         # 止损 = ATR × 倍数
  risk_reward_ratio: 2.0      # 盈亏比（止盈距离 / 止损距离）
```

## 回测引擎兼容性

回测引擎的止损止盈检查逻辑改为：
1. 如果信号带有 `StopLossPrice`/`TargetPrice`（策略计算），使用绝对价格
2. 如果信号没有设置（旧策略），fallback 到全局百分比 `req.StopLossPct`/`req.TakeProfitPct`

这保证了其他策略（箱体、影线、关键位等）的行为不受影响。

## 各策略盈亏比优化（2026-04-17）

### 问题分析

通过 SQL 直接查询 `trade_tracks` JOIN `trading_opportunities` 分析各策略实际盈亏比：

| 策略 | 盈亏比 | 平均盈利 | 平均亏损 | 问题 |
|------|--------|---------|---------|------|
| resistance_break | **2.00** | +103.66 | -51.95 | 唯一盈利 |
| lower_wick_reversal | 1.71 | +54.76 | -31.99 | 盈亏比OK，胜率太低 |
| momentum_bullish | 1.46 | +58.75 | -40.33 | 盈亏比不够补胜率 |
| momentum_bearish | **0.58** | +34.76 | **-60.10** | 亏的比赚的多一倍 |
| upper_wick_reversal | **0.64** | +31.18 | **-48.72** | 同上 |
| support_break | 0.94 | +45.61 | -48.60 | 盈亏比<1 |

**根因分析**（3 个关键 bug）：

1. **Bug 1：移动止损立即激活**（最大元凶）
   `auto_trader.go` 开仓时**没有设置 `TrailingActivationPct`**，config 中也没有这个字段。
   导致 `trailing_stop.go` 中 `activationPrice = entryPrice`，移动止损在入场价就激活。
   效果：价格刚涨 1.5% 就被移动止损平仓，价格跌 2% 才触发固定止损。
   **盈亏比被压缩到约 0.75:1**。

2. **Bug 2：多个策略没有 ATR 止盈止损**
   - key_level：只设止损（0.5% buffer），不设止盈
   - candlestick：只设止损，不设止盈
   - volume：完全没有止盈止损
   没设止盈的策略 → fallback 到全局 5% → 很多时候达不到

3. **Bug 3：opportunity SL/TP 被"最后一个信号"覆盖**
   `opportunity_aggregator.go` 中 `createOpportunity/updateOpportunity` 用最后信号的 SL/TP。
   如果一个 box 信号（有合理 SL/TP）后跟一个 volume 信号（无 SL/TP），box 的 SL/TP 就被覆盖为 nil。

### 修复措施

#### 1. 修复移动止损立即激活
- `config/config.yml`：新增 `trailing_activate_pct: 0.03`（盈利 3% 后才激活）
- `auto_trader.go`：开仓设置 `TrailingActivationPct: t.config.TrailingActivatePct`

#### 2. 各策略统一 ATR 止盈止损

公共函数（`strategy/strategy.go`）：
```go
func CalculateATR(klines []models.Kline, period int) float64 { ... }
func CalculateSLTP(entryPrice, direction, atr, atrMultiplier, riskRewardRatio) (stopLoss, takeProfit float64) { ... }
```

| 策略 | 修复内容 |
|------|---------|
| trend_strategy.go | 已有完整 ATR SL/TP，确认参数合理 |
| key_level_strategy.go | 新增 `CalculateSLTP` 止盈，修复变量作用域 bug |
| candlestick_strategy.go | 新增 `CalculateSLTP` 止盈，修改 `newSignal` 参数 |
| volume_strategy.go | 新增完整 ATR SL/TP，计算后传入两个检测子函数 |

各策略配置文件（`config/config.yml`）新增：
```yaml

  atr_period: 14
  atr_multiplier: 2.0
  risk_reward_ratio: 2.0
candlestick:
  atr_multiplier: 2.0
  risk_reward_ratio: 2.0

  atr_period: 14
  atr_multiplier: 2.0
  risk_reward_ratio: 2.0
```

#### 3. 修复 opportunity SL/TP 覆盖逻辑
- `opportunity_aggregator.go`：`createOpportunity` 和 `updateOpportunity` 改为优先保留已有非空值，nil 不覆盖

```go
// 修复前（last-signal-wins）
for _, sig := range signals {
    if sig.StopLossPrice != nil {
        stopLoss = sig.StopLossPrice  // volume信号的nil会覆盖box的好值
    }
}

// 修复后（first-non-nil-wins）
for _, sig := range signals {
    if stopLoss == nil && sig.StopLossPrice != nil {
        stopLoss = sig.StopLossPrice
    }
    if takeProfit == nil && sig.TargetPrice != nil {
        takeProfit = sig.TargetPrice
    }
}
```

## 反馈闭环排查

### 问题：signal_type_stats 表为空

后端已重启（含 opportunity_id 修复），但 `signal_type_stats` 仍然为空。

**排查结果**：
- `updateSignalTypeStatsAsync` 在 goroutine 中运行，无 recover，panic 会被静默吞掉
- `OpportunityRepoPG.GetByID()` 的 scan 错误被 `return nil, nil` 静默吞掉
- 两个静默吞错误的叠加导致无法定位具体失败原因

**根因定位**（关键 bug）：
`trade_track_repo.go` 的 `tradeTrackColumns` 常量和 `GetOpenPositions` 查询**没有包含 `opportunity_id` 列**。
导致从数据库读出的 `TradeTrack` 对象的 `OpportunityID` 始终为 nil，
反馈闭环在 `updateSignalTypeStatsAsync` 中因 `track.OpportunityID == nil` 直接 return，统计永远无法写入。

**修复措施**：
1. `trade_track_repo.go`：`tradeTrackColumns` 加上 `opportunity_id` 列
2. `trade_track_repo.go`：`scanTradeTrack` 和 `scanTradeTrackWithSymbolCode` 加上 `&track.OpportunityID`
3. `trade_track_repo.go`：`GetOpenPositions` 和 `GetClosedTracks` 的 SELECT 加上 `opportunity_id`
4. `trade_executor.go`：goroutine 加 `defer recover` + panic 日志
5. `trade_executor.go`：`updateSignalTypeStatsAsync` 关键路径加 Debug/Warn 日志
6. 手动补录历史 253 笔已关闭交易到 `signal_type_stats`（364 条统计记录）

**验证结果**：修复后重启服务，5 分钟内 119 条统计记录自动更新，反馈闭环正常工作。
