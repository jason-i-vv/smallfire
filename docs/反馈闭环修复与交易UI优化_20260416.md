## 1. 反馈闭环修复

### 问题描述
交易关闭后，`signal_type_stats` 表没有数据更新。反馈闭环没有工作。

### 根本原因
`trade_track_repo.go` 的 `Create()` 方法 INSERT 语句**没有包含 `opportunity_id` 字段**。

`auto_trader.go` 设置了 `track.OpportunityID = &oppID`，但 `Create()` 的 SQL 没有把这个字段写入数据库，导致所有交易的 `opportunity_id = NULL`。`updateSignalTypeStatsAsync()` 第一行就 early return 了：

```go
if e.statsRepo == nil || track.OpportunityID == nil {
    return
}
```

### 修复
**`internal/repository/trade_track_repo.go`** - 在 `Create()` 方法的 INSERT 语句中添加 `opportunity_id`：

```diff
- INSERT INTO trade_tracks (signal_id, symbol_id, direction, ...)
+ INSERT INTO trade_tracks (signal_id, opportunity_id, symbol_id, direction, ...)
```

同时在 `Repository` 接口中添加 `GetByOpportunityID` 方法。

### 数据清理
```sql
DELETE FROM trade_tracks WHERE opportunity_id IS NULL;
```

### 验证
1. 重启后端 `make dev`
2. 等待 AutoTrader 创建新交易
3. 查询验证：`SELECT id, opportunity_id, symbol_id FROM trade_tracks ORDER BY id DESC LIMIT 5;`
4. 等待平仓后验证：`SELECT COUNT(*) FROM signal_type_stats;`

## 2. 通知评分阈值过滤

### 问题描述
所有交易机会都发送飞书通知，包括评分很低的机会（30-40分），噪音太多。

### 修复
在 `opportunity_aggregator.go` 的 `notifyIfNeeded()` 方法中添加评分阈值检查：

```go
const minScoreToNotify = 60

func (a *OpportunityAggregator) notifyIfNeeded(opp *models.TradingOpportunity) {
    if a.notifier != nil && opp.Score >= a.minScoreToNotify {
        if err := a.notifier.SendOpportunity(opp); err != nil {
            a.logger.Error("发送交易机会通知失败", ...)
        }
    }
}
```

只有评分 >= 60 的机会才会发送通知。

## 4. 入场价等于出场价问题修复

### 问题描述
部分交易记录显示 `entry_price = exit_price`，导致 PnL 接近零（只有手续费损失）。

### 可能原因
1. `SuggestedTakeProfit` 被设置为入场价
2. 价格精度问题导致价格比较出现误差
3. K线数据停止更新，价格提供商返回与入场价相同的当前价

### 修复
在 `ClosePosition()` 方法中添加防御性检查：

```go
func (e *TradeExecutor) ClosePosition(track *models.TradeTrack, reason string, exitPrice float64) error {
    // 防御性检查：避免在入场价平仓（会导致实际亏损 = 手续费）
    if math.Abs(exitPrice-*track.EntryPrice) < 0.00000001 {
        e.logger.Warn("平仓价格与入场价相同，跳过本次平仓",
            zap.Int("track_id", track.ID),
            zap.Float64("entry_price", *track.EntryPrice),
            zap.Float64("exit_price", exitPrice),
            zap.String("reason", reason))
        return nil
    }
    // ... 正常平仓逻辑
}
```

## 4. 入场价等于出场价问题修复

### 问题描述
部分交易记录显示 `entry_price = exit_price`，导致 PnL 接近零（只有手续费损失）。

### 根本原因
信号生成层的 `calculateTarget` 函数在计算止盈价格时，如果 `klineRange`（K线振幅）接近 0，会导致止盈价等于入场价。

**触发条件**：
- 市场数据同步停止，K线停止更新
- K线 High = Low = Close（价格静止）
- 策略算法没有最小波动率保护

**数据流**：
```
K线数据静止 (High ≈ Low ≈ Close)
        ↓
klineRange ≈ 0
        ↓
wick_strategy: currentPrice + klineRange*1.5 ≈ currentPrice
box_strategy: box.HighPrice + width*1.5 ≈ entryPrice
        ↓
TargetPrice ≈ EntryPrice
        ↓
auto_trader 使用该值作为止盈
        ↓
交易开仓后立即触发止盈
```

### 修复
在所有策略的止盈止损计算函数中添加最小波动率检查。

**wick_strategy.go**:
```go
func calculateTarget(kline, direction) {
    klineRange := kline.High - kline.Low
    minRange := currentPrice * 0.005  // 0.5% 最小波动

    if klineRange < minRange {
        // 低波动：使用固定的 1.5% 目标
        return currentPrice * 1.015
    }
    // 正常情况：使用 K线范围倍数
    return currentPrice + klineRange*1.5
}
```

**box_strategy.go, candlestick_strategy.go**: 同样添加最小波动率检查

**key_level_strategy.go**: 止损直接用 `level.Price`（支撑/阻力位价格），没有缓冲。修复为在 level 之外添加 0.5% 缓冲。

### 受影响策略

| 策略 | StopLoss | TargetPrice | 状态 |
|------|----------|-------------|------|
| wick_strategy | buffer = klineRange*0.002 | klineRange*1.5 | ✅ 已修复 |
| box_strategy | buffer = width*0.005 | width*1.5 | ✅ 已修复 |
| candlestick_strategy | buffer = klineRange*0.1 | 无 | ✅ StopLoss已修复 |
| key_level_strategy | level.Price（无缓冲） | 无 | ✅ 已修复 |
| trend_strategy | 无（依赖交易层默认） | 无 | ✅ 无需修复 |
| volume_strategy | 无（依赖交易层默认） | 无 | ✅ 无需修复 |

## 5. 回测信号重复和验证缺失问题

### 问题描述
回测结果显示：
- 65 个信号中，42 个 unique，23 个是重复的（同一 kline_time 出现两次）
- `signal_verification` 中 valid_count = 0，trend 信号全部标记为 "skipped"
- 最后一根 K 线标注了很多买卖平仓标签

### 根本原因

**问题 A - `ptrTime` 指针别名 bug**：返回栈上局部变量地址，导致所有信号 `KlineTime` 可能指向同一内存

**问题 B - 趋势信号验证未实现**：`signal_verifier.go` 只实现了 `candlestick` 策略验证，`trend` 策略全部跳过

### 修复

**`strategy/strategy.go`**：修复 `ptrTime` 返回独立指针
```go
func ptrTime(t time.Time) *time.Time {
    tmp := t // 强制 Go 将 tmp 分配在栈帧稳定位置，避免被复用
    return &tmp
}
```

**`signal_verifier.go`**：添加 `verifyTrendSignal` 函数，验证：
- EMA 三线排列方向与信号方向一致
- EMA 斜率方向与信号方向一致
- 信号价格位于 EMA 附近（5% 以内）

### 验证
- 回测后 `valid_count` 应 > 0（趋势信号通过验证）
- `duplicate_count` 应为 0（`ptrTime` 修复后每信号有独立时间戳）

## 6. 趋势策略逆势开仓问题

### 问题描述
ETHUSDT 2月3日 8:00 开了一笔多单，但当时整体趋势是下跌趋势（bearish）。

### 根本原因
TrendStrategy 的信号生成逻辑和趋势检测逻辑**使用了不同的 EMA 指标**：

- **趋势检测** (`trend_calculator.go`)：比较 EMAShort、EMAMedium、EMALong 三条线的**排列顺序**
  - 多头：`EMAShort > EMAMedium > EMALong`
  - 空头：`EMAShort < EMAMedium < EMALong`

- **信号生成** (`checkPullback`)：只使用被检测的那条 EMA（由调用方传入）的 **10 根 K 线变化率** (`emaSlope > 0` = 开多，`emaSlope < 0` = 开空)

结果：在下跌趋势中的反弹阳线，使短期 EMA 短暂上升（`emaSlope > 0`），触发了做多信号，但整体趋势仍是空头。

### 修复
在 `checkPullback` 中增加**大趋势方向对齐检查**：

```go
// 使用三条EMA的排列判断大趋势方向
if emaSlope > 0 {
    // 新增：大趋势必须是多头
    if trendDirection != models.DirectionLong {
        return nil
    }
    // ... 原有做多逻辑
}

if emaSlope < 0 {
    // 新增：大趋势必须是空头
    if trendDirection != models.DirectionShort {
        return nil
    }
    // ... 原有做空逻辑
}
```

### 验证
- 回测时，2月3日 00:00 的多单（bearish 趋势中做多）将不再出现
- 所有信号方向现在与大趋势方向保持一致
