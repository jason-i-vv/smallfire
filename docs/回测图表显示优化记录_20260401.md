# 回测图表显示优化记录_20260401

## 修复背景
用户反馈前端回测界面中，除了箱体回测外，其他策略（如趋势策略、信号、交易记录）在点击查看图表时显示不正确。主要问题在于：
1. 非箱体策略下依然尝试加载和显示不相关的关键价位（阻力位/支撑位）。
2. 图例显示逻辑不匹配，导致在非相关模式下显示了空的或错误的价格信息。
3. 信号时间戳解析错误，导致 K 线数据获取的时间范围不正确。
4. **K 线数据在未收盘时就被写入数据库，导致收盘价不正确**。

## 修改内容

### 1. KlineChart.vue 逻辑优化
- **数据加载逻辑**：在 `updateKlineData` 函数中，修改了根据 `sourceType` 决定是否调用 `fetchKeyLevels()` 的逻辑。现在只有在 `sourceType` 为 `resistance_break` 或 `support_break`（关键价位相关信号）时才会主动加载关键价位数据。
- **绘制隔离**：对于 `signal` (信号)、`trade` (交易)、`trend` (趋势) 等其他回测类型，现在只执行时间轴滚动定位逻辑，不再默认绘制阻力/支撑位虚线，保持图表简洁且专注当前策略目标。

### 2. 模板图例显示控制
- **关键价位图例**：将 `v-if` 判断条件从「非箱体模式」改为「明确的关键价位突破模式」(`['resistance_break', 'support_break'].includes(sourceType)`)。这避免了趋势策略或普通交易记录点击进入时，上方出现无关的「阻力位/支撑位」统计和图例。

### 3. 修复 K 线数据时间范围获取逻辑
**问题**：`fetchKlines` 函数只在有 `boxStart && boxEnd`（箱体时间范围）时才设置 `start_time` 和 `end_time` 参数。对于信号/交易/趋势类型，没有传递 `boxStart/boxEnd`，导致只获取最近的 500 根 K 线，无法正确定位到信号发生时刻。

**修复**：在 `fetchKlines` 中添加对 `signalTime` 的检测，当有信号时间但没有箱体时间时，以 `signalTime` 为中心计算时间范围：
```javascript
const hasSignalTime = !!(signalTime.value)

if (hasTimeRange) {
  // 箱体模式：使用箱体时间范围
  params.start_time = boxStart.value - 50 * periodSeconds
  params.end_time = boxEnd.value + 50 * periodSeconds
} else if (hasSignalTime) {
  // 信号模式：以信号时间为中心，前后各获取约100根K线
  params.start_time = signalTime.value - 100 * periodSeconds
  params.end_time = signalTime.value + 100 * periodSeconds
}
```

### 4. 修复信号时间戳解析错误
**问题**：原代码使用 `new Date(route.query.signalTime)` 解析时间戳，当 URL 参数是 `1775023789.405` 这样的秒级时间戳（可能是小数）时，`new Date()` 无法正确解析。

**修复**：添加专门的 `parseSignalTime` 函数，正确处理秒级和毫秒级时间戳：
```javascript
const parseSignalTime = (value) => {
  if (!value) return null
  const num = parseFloat(value)
  if (isNaN(num)) return null
  // 如果大于 1e12，说明是毫秒级时间戳
  if (num > 1e12) return num / 1000
  // 否则是秒级时间戳（可能是小数形式）
  return num
}
const signalTime = ref(parseSignalTime(route.query.signalTime))
```

### 5. 修复 K 线数据未收盘就写入数据库的问题
**问题**：在 `sync_service.go` 的 `syncSymbolKlines` 函数中，只检查 K 线是否已存在，不检查是否已收盘。当某个周期（如 1h）还未收盘时（比如当前是 13:30），交易所已经返回这根 K 线的数据，`close_price` 只是当前价格而不是真正的收盘价，导致数据库中的 K 线数据不正确。

**修复**：在创建新 K 线前，检查 K 线是否已经收盘。只有当 `k.CloseTime <= now`（收盘时间已过）时才写入数据库：
```go
// 当前时间，用于判断K线是否已收盘
now := time.Now()

for _, k := range klines {
    // 跳过未收盘的K线：只有当K线的收盘时间已过，才写入数据库
    if k.CloseTime.After(now) {
        s.logger.Debug("跳过未收盘K线", ...)
        continue
    }
    // ... 后续创建逻辑
}
```

**数据库清理**：对于已经写入数据库的未收盘 K 线（`is_closed = true` 但实际未收盘），需要执行清理 SQL：
```sql
-- 删除未收盘的K线数据（收盘时间在当前时间之后的）
DELETE FROM klines
WHERE is_closed = true AND close_time > NOW();
```

1. 在回测页面运行 **箱体突破** 策略，点击箱体列表或信号，应依然能看到完整的箱体绘制和图例。

5. **K 线数据验证**：检查数据库中的 K 线收盘价是否正确，特别是检查最近一根 K 线的 `close_price` 是否等于 `open_time` 对应周期的真正收盘价。

## 20260402 - Info-Bar 信号描述信息展示 & 信号时间修复

### 修复回测信号跳转使用 created_at 而非 kline_time
**问题**：回测页面点击信号跳转到 K 线图时，`signalTime` 参数使用的是 `item.created_at`（信号创建时间）。同一批次的信号创建时间几乎相同，导致不同信号（如 `support_break` 和 `resistance_break`）的 `signalTime` 值几乎一样，无法正确定位到各自的 K 线位置。

**修复**：`Backtest.vue:784` 将 `item.created_at` 改为 `item.kline_time || item.created_at`，优先使用 K 线时间，与表格显示时间列保持一致。

### Info-Bar 信号描述信息展示
**需求**：在 K 线图的 info-bar 区域（周期/K线数 后面），根据当前信号类型展示该信号的关键数据描述，让用户一眼就能看到信号的核心信息。

**实现**：
- 新增 `getSignalDescription(signal)` 函数，替代原来的 `getBreakoutLevelPrice`，统一处理所有信号类型的描述：
  - **关键价位信号**：突破/跌破的价位和距离百分比
  - **箱体信号**：箱顶/箱底价格和突破价
  - **引线信号**：实体占比和前期相似引线数量
  - **假突破信号**：突破点价格
  - **量价信号**：放量倍数和均价
  - **趋势信号**：signal_data 为空，不展示额外描述
- 在 info-bar 中用带主题色半透明背景的标签样式展示描述文字

## 20260402 - 回测信号重复问题分析与修复

### 问题分析
回测结果中 168 个信号看似"重复"，实际分析后并非重复：
- 110 个不同价格 = 110 根不同 K 线触发
- 其中 57 根 K 线各触发 2 个信号（price_surge + volume_surge/volume_price_rise/fall）
- 这是量价策略的正常行为（一根 K 线最多 2 个信号）

但存在两个导致信号"堆在一起"的问题：
1. **`kline_time` 字段缺失** — 旧版代码生成的回测 JSON 中 Signal 模型没有 KlineTime 字段
2. **`created_at` 全部相同** — 回测中所有信号都用 `time.Now()` 作为 CreatedAt，导致排序后所有信号挤在一起

### 修复内容
1. **`volume_strategy.go`** — 将 `KlineTime: latest.CloseTime` 改为 `KlineTime: latest.OpenTime`，使用 K 线开盘时间作为信号时间，与其他策略（key_level 在 backtest_service.go 中）保持一致
2. **`backtest_service.go`** — 信号排序从 `CreatedAt` 改为 `KlineTime`，回测时 CreatedAt 都是同一时刻无法区分先后，用 KlineTime 才能正确反映信号在 K 线时间轴上的顺序

### 关键价位策略同一K线重复信号修复

**问题**：`keyLevelStrategyAnalyzer.Analyze` 遍历所有未突破的阻力位/支撑位，当一根 K 线同时突破多个价位时，每个价位都产生一个独立信号。例如 `2026-03-10T01:00:00Z` 一根 K 线产生了 5 个 `resistance_break` 信号（突破了 5 个不同阻力位）。

**修复原则**：同一根 K 线只取**被突破的距当前价格最近的**价位产生信号，其他价位标记为已突破但不产生额外信号。

**修改文件**：
1. **`backtest_service.go` `keyLevelStrategyAnalyzer.Analyze`** — 阻力位取被突破中价格最高的（最接近当前价），支撑位取被跌破中价格最低的，各只产生一个信号
2. **`key_level_strategy.go` `Analyze`** — 实时策略同样修复，避免多个同方向价位同时突破时产生多个信号

## 20260402 - 关键价位信号描述信息增强

### 需求
信号中需要关联阻力位/支撑位的详细信息，包括为什么这是一个阻力位/支撑位的描述，方便在回测和实盘监测中判断信号质量。

### 实现方案

**回测 `keyLevelStrategyAnalyzer`（backtest_service.go）**：
- `LevelInfo` 新增 `TouchCount` 字段，统计价位被后续 K 线触及但未突破的次数
- 新增 `countTouches` 方法，遍历价位形成后的 K 线，统计价格接近但未突破的次数
- 信号 `signal_data` 新增字段：
  - `level_price` — 价位价格
  - `level_time` — 价位形成时间
  - `level_distance` — 突破/跌破幅度百分比
  - `touch_count` — 被触及次数
  - `klines_away` — 价位形成距离突破了多少根 K 线
  - `level_description` — 中文描述，如 "波峰阻力位，触及5次，形成于12根K线前"

**实盘 `KeyLevelStrategy`（key_level_strategy.go）**：
- 利用已有的 `LevelSubtype`（current_high/prev_high/current_low/prev_low）和 `KlinesCount`
- 信号 `signal_data` 新增 `level_description` 字段：
  - 如 "近期高点阻力位，触及3次，距突破1.25%"
  - 如 "前期低点支撑位，触及5次，距跌破2.10%"

**前端 `getSignalDescription`（KlineChart.vue）**：
- 优先使用 `level_description` 展示，兼容旧数据无此字段的情况
