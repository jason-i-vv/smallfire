## 核心概念：三层模型

```
Strategy → Signal → Trading Opportunity
```

| 层级 | 说明 | 举例 |
|------|------|------|
| Signal | 单一策略对单一标的的判断 | key_level 说 BTCUSDT 突破阻力位，看多 |
| Trading Opportunity | 多个 Signal 在同一标的上的聚合 | key_level 看多 + wick 看多 + candlestick 看多 = BTCUSDT 做多机会 |
| Trade | 对一个 Opportunity 执行交易 | 对 BTCUSDT 做多半仓，止损 $94,500 |

**Signal 是原材料，Opportunity 是成品。** 所有下游行为（展示、通知、交易、回测统计）都应该基于 Opportunity，而不是直接基于 Signal。

## Problem Statement

当前星火量化的 6 个策略会持续产生交易信号，但不是每个信号都有交易价值。用户每天会面对 20-50 个信号，需要：
1. 自动评估信号质量，过滤掉噪声信号
2. 对高质量信号自动执行交易，低质量信号需要用户确认
3. 交易完成后反馈到评分系统，让系统越跑越准

## 核心洞察

**信号和交易是两件事。** 信号是策略说"这里可能有机会"，交易是"我决定把钱放进去"。中间差的就是信号质量评估。

量化交易的基本认知：**永远无法在交易前 100% 确定一个信号是否赚钱。** 我们需要的是一个概率框架，用评分表达"这个信号历史上赢的概率是 X%"，然后用户设置阈值来决定自动交易的门槛。

## 信号评分模型

评分范围 0-100，由 5 个维度加权组成：

| 维度 | 权重 | 数据来源 | 说明 |
|------|------|---------|------|
| 策略历史胜率 | 30% | signal_type 的历史交易统计 | 该信号类型过去 30 天的胜率，如 "resistance_break" 胜率 35% |
| 多策略共识 | 25% | 同一时间窗口内其他策略的信号 | 3+ 个独立策略同时看多 = 强共识 |
| 信号强度 | 20% | signal.strength | 策略自身计算的强度（1-5），已包含形态质量、关键位等 |
| 成交量确认 | 15% | 信号产生时及之前几根 K 线的成交量 | 放量配合 > 1.5 倍均量 = 有量确认 |
| 市场状态匹配 | 10% | EMA 趋势状态 vs 策略类型 | 趋势市中趋势策略加分，震荡市中反转策略加分 |

### 评分计算

```
score = (strategyWinRate * 0.30) +
        (confluenceScore * 0.25) +
        (strengthNormalized * 0.20) +
        (volumeConfirm * 0.15) +
        (regimeMatch * 0.10)
```

各维度归一化到 0-100：
- **strategyWinRate**: `(30天该信号类型胜率 / 100) * 100`
- **confluenceScore**: 
  - 1 个策略同意: 20
  - 2 个策略同意: 50
  - 3 个策略同意: 80
  - 4+ 个策略同意: 100
- **strengthNormalized**: `(strength / 5) * 100`
- **volumeConfirm**: 
  - 无量确认: 30
  - 1-1.5x 均量: 60
  - 1.5x+ 均量: 100
- **regimeMatch**: 
  - 趋势市 + 趋势策略或震荡市 + 反转策略: 100
  - 趋势市 + 反转策略或震荡市 + 趋势策略: 40
  - 横盘: 20

### 多策略共识的具体规则

只在**同一时间窗口内**（前后 3 根 K 线）检查多策略共识：
- key_level 的 resistance_break（做多）+ wick 的 lower_wick_reversal（做多）+ candlestick 的 engulfing_bullish（做多）= 3 个独立策略共识，高分
- 策略类型必须不同才算独立（同一策略的多个信号不重复计数）

## 交易决策层

评分产生后，根据用户配置的阈值决定行为：

| 评分区间 | 行为 | 前端显示 |
|---------|------|---------|
| >= 80 | 自动交易，全仓位 | 绿色"强烈推荐" |
| 60-79 | 自动交易，半仓位 | 蓝色"建议交易" |
| 40-59 | 提示用户，等待确认 | 黄色"观望" |
| < 40 | 不交易，仅记录 | 灰色"不推荐" |

用户可在设置页面配置：
- 自动交易阈值（默认 60）
- 每笔最大仓位比例
- 每日最大交易次数
- 单标的最大持仓数

## 反馈闭环

每笔交易平仓后，更新对应信号类型的历史统计：

```
signal_type: "resistance_break"
direction: "long"
period: "1h"
last_30d_stats: {
  total_trades: 45,
  win_rate: 0.356,
  avg_return: 0.023,
  profit_factor: 1.45,
  optimal_stop_loss: 0.018,  // 该信号类型历史最优止损
  optimal_take_profit: 0.042  // 该信号类型历史最优止盈
}
```

这些统计反馈到评分模型中，使 `strategyWinRate` 维度持续更新。

## 数据模型

### 核心新实体：trading_opportunities

```sql
CREATE TABLE trading_opportunities (
    id SERIAL PRIMARY KEY,
    symbol_id INTEGER NOT NULL,
    symbol_code VARCHAR(50) NOT NULL,
    direction VARCHAR(10) NOT NULL,         -- long / short
    score INTEGER NOT NULL,                -- 综合评分 0-100
    signal_ids INTEGER[] NOT NULL,          -- 组成此 opportunity 的 signal ID 列表
    signal_types VARCHAR(50)[] NOT NULL,    -- 参与的策略类型列表
    strategy_count INTEGER DEFAULT 0,        -- 参与的独立策略数量
    consensus_direction VARCHAR(10),          -- 多策略共识方向
    recommended_action VARCHAR(20),          -- auto_trade / manual_review / skip
    recommended_position_size NUMERIC(4,2), -- 建议仓位比例
    stop_loss_price NUMERIC(18,8),
    target_price NUMERIC(18,8),
    score_details JSONB,                 -- 各维度评分明细
    status VARCHAR(20) DEFAULT 'active',   -- active / expired / traded / expired_traded
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    traded_at TIMESTAMP,
    expired_at TIMESTAMP
);

CREATE INDEX idx_opportunities_symbol ON trading_opportunities(symbol_id, status);
CREATE INDEX idx_opportunities_score ON trading_opportunities(score DESC, created_at DESC);
```

**Opportunity 生命周期**：

```
信号产生 → 聚合检查(同标的+同方向) → 创建 Opportunity
    ↓
每根新K线 → 更新有效性 → 评分变化 → 状态可能变化
    ↓
评分 >= 阈值 → traded        评分衰减/信号过期 → expired
    ↓
交易平仓 → 更新 signal_type_stats → Opportunity 标记 completed
```

**聚合规则**：
- 每根新 K 线，策略执行器产生信号后，触发聚合检查
- 检查当前标的的所有 active signals，按方向分组
- 同方向有 2+ 个信号时，创建一个 Opportunity
- Opportunity 创建后，参与聚合的 signals 标记为 `status: absorbed`（不再独立展示）
- Opportunity 中的 signals 过期后，如果剩余信号 < 2 个，Opportunity 标记 expired

### 修改表：signal_type_stats

每个信号类型（如 resistance_break、lower_wick_reversal）的历史交易统计，驱动评分中的 `strategyWinRate` 维度。

```sql
CREATE TABLE signal_type_stats (
    id SERIAL PRIMARY KEY,
    signal_type VARCHAR(50) NOT NULL,       -- 信号类型: resistance_break, lower_wick_reversal 等
    direction VARCHAR(10) NOT NULL,          -- long / short
    period VARCHAR(10) NOT NULL,             -- K线周期: 1h, 4h, 1d
    symbol_id INTEGER,                        -- 标的ID（NULL=全局统计，非NULL=标的级别统计）
    total_trades INTEGER DEFAULT 0,           -- 总交易次数
    win_count INTEGER DEFAULT 0,             -- 盈利次数
    loss_count INTEGER DEFAULT 0,            -- 亏损次数
    win_rate NUMERIC(6,4) DEFAULT 0,         -- 胜率
    avg_return NUMERIC(10,6) DEFAULT 0,      -- 平均收益率
    profit_factor NUMERIC(10,4) DEFAULT 0,   -- 盈亏比
    optimal_stop_loss NUMERIC(8,6),          -- 该信号类型历史最优止损
    optimal_take_profit NUMERIC(8,6),        -- 该信号类型历史最优止盈
    stats_window_days INTEGER DEFAULT 30,    -- 统计窗口（天）
    last_trade_at TIMESTAMP,                 -- 最近一次交易时间
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(signal_type, direction, period, symbol_id)
);

CREATE INDEX idx_signal_type_stats_lookup ON signal_type_stats(signal_type, direction, period);
```

**统计更新逻辑**：每笔交易平仓后，更新对应 signal_type + direction + period 的统计行。同时维护全局统计（symbol_id=NULL）和标的级别统计（symbol_id=X）。

用户可配置的自动交易参数：


## 涉及的模块

### 后端
1. **signal_scorer.go** (新增): 信号评分引擎核心逻辑
   - `ScoreSignal(signal, context) -> ScoreResult`
   - 计算各维度分数，返回总分和各维度明细

2. **signal_type_stats_repo.go** (新增): 信号类型历史统计的数据层
   - `GetStats(signalType, direction, period, symbolId) -> Stats`
   - `UpdateStats(signalType, direction, period, symbolId, tradeResult)`

3. **trading/executor.go** (修改): 交易执行器增加评分检查
   - 收到信号后先评分
   - 根据阈值决定自动交易/提示/忽略

4. **strategy/runner.go** (修改): 策略执行器增加共识检测
   - 每轮分析结束后，检查当前所有策略的信号方向
   - 将共识信息写入信号的 `SignalData`

5. **handler/** (修改): API 层
   - 信号列表接口返回 `score` 和各维度明细
   - 新增 `GET /api/v1/signal-stats` 接口查询历史统计
   - 新增 `PUT /api/v1/trading/config` 接口更新交易配置

### 前端
1. **信号列表页**: 每条信号显示评分徽章（颜色编码）和评分理由
2. **交易设置页**: 自动交易开关、阈值滑块、仓位控制
3. **信号统计面板**: 每个信号类型的历史胜率、盈亏比、最优止损止盈

## 前端展示：币对卡片列表

### 展示范式转变

**当前**：按时间排列的扁平信号列表（每行一个信号）。用户看到的是一堆独立信号，很难判断哪个币对当前最值得交易。

**目标**：按币对聚合的卡片列表。每个币对是一张卡片，展示当前所有有效信号的共识和评分。类似交易所的"自选交易对"列表。

### 信号时效性与聚合逻辑

不同类型的信号有不同的有效期。聚合时只取仍在有效期内的信号：

| 信号类型 | 有效期 | K线数(1H) | 过期后行为 |
|---------|--------|-----------|----------|
| wick (引线反转) | 短期 | 5 根 | 从卡片中移除该信号 |
| candlestick (K线形态) | 短期 | 3 根 | 从卡片中移除该信号 |
| key_level (关键位突破) | 中期 | 20 根 | 突破失效后移除 |
| volume_price (量价分析) | 中期 | 15 根 | 量能信号衰减后移除 |
| trend (趋势跟踪) | 长期 | 直到反转 | 趋势反转后移除 |
| box (箱体突破) | 长期 | 直到破位 | 箱体破位后移除 |

**每根新 K 线到达时**，策略执行器更新所有活跃信号的有效性状态，过期的信号标记为 `expired`，不再参与评分和展示。

### 排序规则

卡片列表按综合评分从高到低排序：
1. 评分最高的币对在最前面（最值得交易）
2. 有多策略共识的币对排在前面
3. 同分币对按最近信号产生时间倒序
4. 无有效信号的币对不显示（或放在底部灰色区域）

### 交互流程

1. **默认视图**：卡片列表，每个币对一张卡片
2. **点击卡片**：展开显示该币对的所有信号详情（现有信号表格）
3. **筛选/排序**：按方向（做多/做空）、评分范围、策略类型筛选
4. **设置入口**：自动交易阈值、仓位控制

### 数据模型补充

信号模型新增有效期相关字段：

```go
type Signal struct {
    // ... 现有字段 ...

    // 评分相关
    Score           int       `json:"score"`           // 综合评分 0-100
    ScoreDetails    *JSONB     `json:"score_details"`    // 各维度评分明细
    ValidUntil      *time.Time `json:"valid_until"`      // 信号有效期截止时间
    Status          string    `json:"status"`          // 新增 "expired" 状态
    ConfluenceInfo  *JSONB     `json:"confluence_info"`  // 多策略共识详情
}
```

有效期计算逻辑在策略执行器的 `analyze` 方法中：
- wick/candlestick: `validUntil = klineTime + validity_klines * period`
- key_level: `validUntil = 无固定期限，突破失效时标记 expired`
- trend: `validUntil = 无固定期限，趋势反转时标记 expired`

## AI Agent 增强判定

### 核心原则：信息隔离

AI Agent 进行模糊判定时，**必须屏蔽程序已有的判定结果**，只提供原始市场数据，让 AI 独立得出结论。避免 AI 受到公式评分的锚定效应影响。

**传入 AI 的信息**（原始数据）：
- 标的：BTCUSDT
- 当前价格、24h 涨跌幅
- 最近 N 根 K 线的开高低收成交量
- 各策略产生的信号列表（仅信号类型、方向、产生时间，不含评分）
- 当前持仓状态（有/无、方向、入场价）

**禁止传入 AI 的信息**：
- 公式评分结果
- 各维度评分明细
- 程序的"建议操作"
- 策略胜率等历史统计数据（避免锚定）

### AI 角色定义

```yaml
ai_judge:
  model: "claude-sonnet-4-6"           # 速度优先
  max_tokens: 500                       # 控制输出长度
  temperature: 0.3                      # 低随机性，保持一致性
  trigger:
    score_range: [50, 79]              # 公式评分在 50-79 之间时触发
    max_daily_calls: 50                 # 每日最大调用次数
    cooldown_minutes: 30               # 同一标的冷却时间

  system_prompt: |
    你是一名专业的加密货币交易分析师。你的职责是基于原始市场数据，
    对当前的交易机会给出独立的判断。

    规则：
    1. 你只能基于提供给你的市场数据进行分析
    2. 你不能参考任何历史评分或统计结论
    3. 你的分析必须独立、客观，不受任何已有结论的影响
    4. 你必须给出明确的方向判断和置信度

    输出格式（严格 JSON）：
    {
      "direction": "long" | "short" | "neutral",
      "confidence": 0-100,
      "reasoning": "一句话核心逻辑",
      "risk_level": "low" | "medium" | "high",
      "suggested_stop_loss_pct": 1.5,
      "suggested_take_profit_pct": 3.0
    }
```

### AI 判定与公式评分的融合

```go
func FinalScore(formulaScore int, aiResult *AIJudgment) int {
    if aiResult == nil {
        return formulaScore  // 不在模糊区间，直接用公式分
    }

    // AI 方向与信号方向一致 → 加分
    // AI 方向与信号方向矛盾 → 减分
    // AI 给出 neutral → 轻微减分
    adjustment := 0
    if aiResult.Direction == signalDirection {
        adjustment += int(float64(aiResult.Confidence) * 0.15)  // 最多 +15
    } else if aiResult.Direction == "neutral" {
        adjustment -= 5
    } else {
        adjustment -= int(float64(aiResult.Confidence) * 0.15)  // 最多 -15
    }

    final := formulaScore + adjustment
    if final > 100 { final = 100 }
    if final < 0 { final = 0 }
    return final
}
```

## 实施顺序

### Phase 1: 评分引擎 + 手动展示（不自动交易）
1. 实现 `signal_scorer.go` 核心评分逻辑
2. 实现 `signal_type_stats_repo.go` 数据层
3. 修改策略执行器，在信号中附加共识信息
4. 信号列表 API 返回评分
5. 前端信号列表显示评分徽章

### Phase 2: 反馈闭环 + AI 每日简报
1. 交易平仓后更新 `signal_type_stats`
2. 评分引擎从统计表读取实时胜率
3. 实现 AI 每日市场简报（每天 1 次 LLM 调用）
4. 前端信号统计面板

### Phase 3: 自动交易 + AI 模糊判定
1. 交易执行器接入评分阈值
2. 实现 AI 模糊判定（评分 50-79 时调用，信息隔离）
3. 前端交易设置页面
4. 仓位控制逻辑

1. 信号共识的时间窗口多大？（当前设计是前后 3 根 K 线，需要根据回测调优）
2. 评分权重是否应该允许用户自定义？（当前设计支持，但默认值需要回测确定）
3. 是否需要按市场状态（牛市/熊市/震荡市）自动调整阈值？

## Success Criteria

1. 每个信号都能看到 0-100 的评分和各维度明细
2. 信号统计面板能展示每个信号类型的历史胜率和盈亏
3. 用户设置阈值后，评分 < 阈值的信号不产生交易
4. 交易结果能反馈到统计表，30 天后统计足够稳定

## Next Steps

Phase 1 是最小可用版本，建议先实现评分引擎和历史统计，在前端展示评分但不接入自动交易。跑 2-4 周积累数据后，再开启自动交易功能。

### 通知优化记录（2026-04-12）

#### 变更说明

将飞书通知从"信号级别"改为"交易机会级别"。每个单独信号不再发送飞书通知，只有当多个信号聚合为交易机会（Trading Opportunity）时才发送通知。

#### 原因

策略分析器每轮可能产生大量信号（6种策略 × 多标的 × 多周期），导致飞书通知过于频繁，信息轰炸严重。交易机会是信号的聚合，只有评分达标（≥45分）或信号数≥2个时才创建，质量更高、频率更低。

#### 后端修改文件

| 文件 | 说明 |
|------|------|
| `internal/service/strategy/runner.go` | 移除 `DependencyNotifier` 接口和信号级别的 `SendSignal` 调用 |
| `internal/service/scoring/opportunity_aggregator.go` | 新增 `OpportunityNotifier` 接口，创建交易机会后调用 `SendOpportunity` |
| `internal/service/notification/notifier.go` | `Notifier` 接口新增 `SendOpportunityNotification` 方法 |
| `internal/service/notification/feishu.go` | 实现 `SendOpportunityNotification`，格式化交易机会信息（标的、方向、评分、建议入场/止损/止盈、共识信号） |
| `internal/service/notification/manager.go` | 新增 `SendOpportunity` 方法，路由到所有通知渠道 |
| `cmd/server/main.go` | 依赖注入调整：runner 不再传入 notifier，aggregator 传入 notifyManager |

#### 通知内容格式

交易机会通知包含以下信息：
- 标的代码和市场标签（加密/A股/美股）
- 方向（做多/做空）
- 综合评分（0-100）
- 信号数量
- 建议入场价、止损价、止盈价
- 共识信号列表（各策略信号类型）

#### 数据流变更

```
# 之前：每个信号都发通知
策略分析 → 产生信号 → SendSignal() → 飞书通知（频繁）

# 现在：只有交易机会才发通知
策略分析 → 产生信号 → 聚合检查 → 创建/更新 TradingOpportunity → SendOpportunity() → 飞书通知（精准）
```
