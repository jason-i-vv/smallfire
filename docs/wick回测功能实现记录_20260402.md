# Wick影线回测功能实现记录

## 日期
2026-04-02

## 背景
Wick影线策略已实现并在实时行情中正常工作，但回测功能未集成。前端已支持选择 wick 策略，但后端验证会拒绝该策略类型，且回测分析器未针对 wick 策略做专门适配。

## 问题分析

### 1. 后端验证缺失
- `internal/handler/backtest_handler.go` 的 `validStrategies` 映射中未包含 `"wick"`
- 前端选择 wick 策略发起回测时，请求被 400 拒绝

### 2. 回测分析器缺失
- wick 策略的 `Analyze` 方法依赖 `deps.TrendRepo.GetActive()` 和 `deps.LevelRepo.GetActive()` 获取趋势和关键位数据
- 在回测环境中，这些数据源不可用（回测使用历史K线滑动窗口，没有实时数据库数据）
- 需要创建专用的回测分析器，在回测中自行计算趋势和关键位

### 3. 前端信号标签缺失
- `Backtest.vue` 的 `getSignalTypeLabel` 映射中未包含 wick 相关的信号类型
- 回测结果的信号列表中，wick 信号会显示原始英文字段名

## 修改内容

### 1. `internal/handler/backtest_handler.go`
- 在 `validStrategies` 中添加 `"wick": true`

### 2. `internal/service/backtest/backtest_service.go`
- 新增 `wickStrategyAnalyzer` 结构体，实现 `strategyAnalyzer` 接口
- 自行实现趋势计算（基于30根K线简单均线）
- 自行实现假突破检测（基于最近20根K线高低点）
- 自行实现引线形态检测（复用 wick 策略的检测逻辑）
- 自行计算信号强度、止盈止损、目标价格
- 在 `newStrategyAnalyzer` 中添加 `case "wick"` 分支

### 3. `starfire-frontend/src/views/backtest/Backtest.vue`
- 在 `getSignalTypeLabel` 中添加 wick 信号类型映射：
  - `upper_wick_reversal` -> 上引线反转
  - `lower_wick_reversal` -> 下引线反转
  - `fake_breakout_upper` -> 假突破上引线
  - `fake_breakout_lower` -> 假突破下引线

## 回测分析器设计说明

### wickStrategyAnalyzer 核心逻辑
1. **引线形态检测**：与实时策略保持一致，通过实体占比、引线/实体比例判断上/下引线
2. **趋势计算**：使用最近30根K线的简单均线判断趋势方向（uptrend/downtrend/sideways）
3. **假突破检测**：基于最近20根K线的高低点，判断当前K线是否突破后回落
4. **信号强度**：基于趋势强度、假突破、形态明显程度、历史相似形态综合计算（1-5）
5. **止盈止损**：止损设在引线端点外侧（0.2%缓冲），目标价为K线范围的1.5倍

### 与实时策略的差异
- 实时策略通过 `TrendRepo` 和 `LevelRepo` 获取趋势和关键位数据
- 回测分析器通过K线数据自行计算，不依赖数据库
- 回测分析器暂未实现关键位检测（支撑/阻力位），后续可根据需要补充

## 2026-04-03 统一实盘和回测分析逻辑

### 背景
回测中 wick 策略使用了独立的 `wickStrategyAnalyzer` 实现，与实盘 `WickStrategy` 的分析逻辑不一致，导致实盘能看到十字星信号但回测看不到。核心问题是回测维护了一套独立的分析代码，没有复用实盘策略。

### 问题分析

1. **趋势计算不一致**：实盘从 `TrendRepo.GetActive()` 读取趋势，回测用30根K线均线自行计算
2. **关键位缺失**：实盘调用 `getNearbyKeyLevels()` 获取关键位加成，回测完全没有该逻辑
3. **趋势过期判断**：实盘用 `time.Now()` 判断趋势是否过期，在回测历史数据场景下语义不正确
4. **信号强度差异**：实盘包含关键位加成（+1 strength），回测缺少该项
5. **similarCount 硬编码**：回测中回看窗口硬编码为10根，实盘使用配置 `StrengthLookback`

### 修改内容

#### 1. `internal/service/strategy/wick_strategy.go`

- **`getCurrentTrend` 方法**：签名增加 `klines []models.Kline` 参数，优先从数据库获取趋势，当 `TrendRepo` 为 nil、查询失败、无数据或趋势过期时，回退到 `calculateTrendFromKlines` 自行计算
- **新增 `calculateTrendFromKlines` 方法**：移植回测的均线计算逻辑（30根K线均线 + 偏离度判断），使用 `models.TrendType*` 常量替代硬编码字符串
- **`getNearbyKeyLevels` 方法**：当 `deps.LevelRepo` 为 nil 时直接返回默认值，支持无数据库模式
- **新增 `buildDescription` 方法**：从回测的 `wickStrategyAnalyzer.buildDescription` 移植，用于生成信号描述
- **`Analyze` 方法**：传入 klines 给 `getCurrentTrend`，调用 `buildDescription` 设置信号 `Description` 字段

#### 2. `internal/service/backtest/backtest_service.go`

- **`newStrategyAnalyzer` 函数**：移除 `"wick"` case，改走 `default` 分支使用 `genericStrategyAnalyzer`，直接委托给实盘 `WickStrategy.Analyze`
- **删除独立实现**：移除 `wickStrategyAnalyzer`、`wickFakeBreakout` 结构体及其所有方法（约370行代码），包括 `Analyze`、`buildDescription`、`detectWickType`、`calculateTrend`、`detectFakeBreakout`、`calculateStrength`、`calculateStopLoss`、`calculateTarget`、`buildSignalData`、`GetBoxes`、`GetTrends`

### 统一后的架构

```
回测调用链：
  BacktestService.RunBacktest()
    -> strategyFac.GetStrategy("wick")  // 返回 WickStrategy 实例（带真实 deps）
    -> newStrategyAnalyzer(delegate, "wick")
    -> genericStrategyAnalyzer.Analyze()  // 直接委托给 delegate
      -> WickStrategy.Analyze()
        -> getCurrentTrend(klines)  // 数据库不可用则 calculateTrendFromKlines
        -> getNearbyKeyLevels()     // LevelRepo 为 nil 则跳过
        -> checkReversalSignal()
```

实盘和回测现在共用同一套 `WickStrategy` 代码，确保分析逻辑完全一致。
