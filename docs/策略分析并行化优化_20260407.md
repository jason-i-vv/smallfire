# 策略分析并行化与整点对齐优化

## 时间
2026-04-07

## 背景

15m K 线信号延迟高达 25 分钟（22:00 K 线，22:25:25 才触发），原因有三：
1. **K 线同步串行**：200 个 symbol × 2 个周期 = 400 次串行网络请求
2. **策略执行不与时钟对齐**：`time.NewTicker(5min)` 从启动时间算起，最坏等待 5 分钟
3. **策略分析串行**：单 goroutine 逐 symbol 逐 period 分析

## 改动内容

### 1. Runner 整点对齐执行
- `runner.go`: `runAnalysisLoop` 从 `time.NewTicker` 改为计算下一个整点时间
- 5 分钟周期 → 在 `:00, :05, :10, :15...` 触发
- 每次执行完后重新计算等待时间，确保始终在整点触发

### 2. 策略分析并行化
- `runner.go`: `analyzeAllSymbols` 拆分为 `analyzeAllSymbols` + `analyzeSymbol`
- 按 symbol 并发执行，使用 semaphore（buffered channel）控制并发数
- 默认最大并发 20，通过配置 `max_concurrent_analysis` 可调

### 3. K 线同步并行化
- `sync_service.go`: `runSyncLoop` 中 symbol-period 循环改为并发
- 使用 `sync.Map` + `atomic` 进行并发安全的统计
- 并发数通过配置 `max_concurrent_sync` 可调（Bybit 默认 10，A 股/美股默认 5）

### 4. 新增配置项
```yaml
# config.yml
strategies:
  runner_interval: 300            # 策略执行间隔（秒），整点对齐
  max_concurrent_analysis: 20     # 最大并发分析数

markets:
  bybit:
    max_concurrent_sync: 10
  a_stock:
    max_concurrent_sync: 5
  us_stock:
    max_concurrent_sync: 5
```

## 修改文件
- `internal/config/config.go` - 新增 `RunnerInterval`、`MaxConcurrentAnalysis`、`MaxConcurrentSync` 配置字段
- `config/config.yml` - 新增配置项
- `internal/service/strategy/runner.go` - 整点对齐 + 并行分析
- `internal/service/market/sync_service.go` - 并行同步
- `cmd/server/main.go` - 传递新配置参数

## 预期效果
- **信号延迟**：从 25 分钟级降低到 1-2 分钟级（5 分钟整点周期 + 并行分析秒级完成）
- **K 线同步耗时**：从分钟级降低到秒级

## 补充：飞书通知合并批处理（2026-04-08）

并行化后多个 goroutine 同时产生信号，每个都直接调飞书 API 导致 `too many request` 限流。

### 改动内容
- 新增 `internal/service/notification/batcher.go` — `SignalBatcher` 批处理器
- `Manager.SendSignal` 改为将信号投递到批处理器，而非直接发送
- 批处理器在 3 秒窗口内收集信号，合并为一条汇总消息发送
- 单批最大 50 条信号，超出立即发送

### 合并消息格式
多条信号合并为飞书卡片消息，包含：
- 标题：`🔔 信号汇总 (N)`
- 每条信号的币对、类型、周期、方向、强度、价格

### 新增文件
- `internal/service/notification/batcher.go` - 信号批处理器
