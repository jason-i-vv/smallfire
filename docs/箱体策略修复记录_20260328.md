# 箱体策略修复记录

## 修复日期
2026-03-28

## 问题描述

### 问题1：部分箱体完全不成立，里面的K线几乎是单边趋势
### 问题2：部分箱体没有延续性，明明后面的K线是可以扩展进去的，结果箱体直接结束了
### 问题3：箱体列表中还有重复的箱体

## 根本原因分析

### 问题1根因
旧策略使用**相邻两个Swing点（一个波峰+一个波谷）**构建箱体：
```
波峰[i] ──────► K线只覆盖 i到j (单边下跌/上涨)
波谷[j] ──────►
```
这本质上是**一段趋势**，不是震荡箱体。

### 问题2根因
策略没有"箱体扩展"逻辑：
- 箱体创建后 `KlinesCount` 和 `StartTime` 固定
- 新K线到来时不会检查"是否仍在箱体范围内"并自动延伸

### 问题3根因
两个问题叠加：
1. `boxKey` 使用了 `KlinesCount`，导致同一价格区间但不同K线数的箱体被视为不同箱体
2. `Analyze` 中只检查 `activeBoxes`，没有检查 `boxes` 列表

## 修复内容

### 1. 修复箱体构建逻辑 (`strategy/box_strategy.go`)

#### detectBoxes 改为滑动窗口方式
```go
// 从每个起始 Swing 点出发，向后扩展，寻找最长的有效箱体
for start := 0; start <= len(swings)-4; start++ {
    for end := start + 3; end < len(swings); end++ {
        window := swings[start : end+1]
        box := s.buildBoxFromSwingRange(symbolID, period, window, klines)
        if box == nil { continue }
        if s.isValidBox(box, window) {  // 新增验证
            allBoxes = append(allBoxes, box)
        }
    }
}
```

#### 新增 isValidBox 验证震荡条件
- 检查高点和低点不能是单调递增/递减
- 要求至少1个高点+1个低点触及边界

#### 新增 deduplicateBoxes 同次检测内去重
- 使用包含重叠度（重叠区间占较小箱体宽度的比例）
- 保留K线数最多的箱体

#### 新增 tryExtendBox 箱体延续逻辑
```go
func (s *BoxStrategy) tryExtendBox(box *models.Box, latestKline models.Kline) {
    buffer := box.WidthPrice * s.config.BreakoutBuffer
    if latestKline.ClosePrice >= box.LowPrice-buffer &&
       latestKline.ClosePrice <= box.HighPrice+buffer {
        box.KlinesCount++
        box.UpdatedAt = time.Now()
        s.deps.BoxRepo.Update(box)
    }
}
```

#### 修复 calculateOverlap 为 calculateContainmentOverlap
- 原实现：小箱体被大箱体包含时重叠度会被低估
- 新实现：重叠区间占较小箱体宽度的比例

### 2. 修复回测服务 (`backtest/backtest_service.go`)

#### 重写 boxStrategyAnalyzer 的箱体检测逻辑
- 同样使用滑动窗口多Swing点聚合
- 添加 `isValidBox`、`isMonotone`、`deduplicateBoxes` 等辅助函数

#### 修复 boxKey 函数
```go
// 使用 %.0f 避免浮点精度问题
func boxKey(box *models.Box) string {
    return fmt.Sprintf("%.0f_%.0f", box.HighPrice, box.LowPrice)
}
```

#### 增强 Analyze 方法中的去重逻辑
```go
for _, box := range newBoxes {
    key := boxKey(box)
    if _, exists := a.activeBoxes[key]; !exists {
        // 检查是否在 a.boxes 中已存在相同价格区间的箱体
        alreadyExists := false
        for _, existingBox := range a.boxes {
            if boxKey(existingBox) == key {
                alreadyExists = true
                break
            }
        }
        if !alreadyExists {
            a.activeBoxes[key] = box
            a.boxes = append(a.boxes, box)
        }
    }
}
```

## 修复后策略识别箱体的完整流程

```
K线数据
   ↓
detectSwingPoints()        检测波峰波谷（需超过 width_threshold 幅度）
   ↓
滑动窗口 [start..end]       至少4个 Swing 点一组
   ↓
buildBoxFromSwingRange()   **关键修复：箱体边界直接取窗口内所有K线的实际极值**
                          - boxHigh = max(K线的HighPrice)
                          - boxLow = min(K线的LowPrice)
                          - 这样单边行情不可能被识别为箱体
   ↓
isValidBox()               验证：
                          - 高低点非单调（震荡特征）
                          - 触及边界次数满足要求
                          - 任何一根K线超出边界则拒绝
   ↓
deduplicateBoxes()         同次检测内按包含重叠度去重
   ↓
filterValidBoxes()         宽度 >= width_threshold，K线数在 [min,max] 范围内
   ↓
Analyze 中双重去重          基于 activeBoxes 和 boxes 列表去重
   ↓
入库 + tryExtendBox()       活跃箱体每轮自动延续
   ↓
checkBreakout()            收盘价超出边界 + buffer → 生成突破信号
```

### 4. 修正箱体边界计算逻辑（第四轮修复）

**问题**：CYSUSDT 箱体 high=0.7717, low=0.5294，但K线数据从0.77跌到0.33，这是单边下跌趋势，不是震荡箱体。

**根本原因**：`buildBoxFromSwingRange` 用 Swing 点中的价格来确定边界，但这会把单边行情开始时的高点也包含进来，导致边界被不合理地放大。

**正确做法**：箱体边界直接使用窗口内所有K线的实际极值：
```go
// 关键修复：箱体边界直接使用窗口内所有K线的实际极值
boxHigh := boxKlines[0].HighPrice
boxLow := boxKlines[0].LowPrice
for _, k := range boxKlines {
    if k.HighPrice > boxHigh {
        boxHigh = k.HighPrice
    }
    if k.LowPrice < boxLow {
        boxLow = k.LowPrice
    }
}
```

这样，单边行情（如从0.77跌到0.33）不可能被识别为箱体，因为边界本身就是K线的实际极值。

## 第五轮修复：波动率检查过滤单边行情

### 问题
虽然修复了边界计算，但单边行情（如从0.77跌到0.33的CYSUSDT）仍可能被识别为箱体，因为：
- 边界从K线极值计算，K线都在边界内
- 单调性检查对非严格单调趋势无效

### 解决方案
添加波动率检查：`volatilityRatio = totalMovement / boxWidth`

- **单边行情**：totalMovement ≈ boxWidth，比率接近 1.0 或更高
- **震荡行情**：价格来回波动，totalMovement > boxWidth，比率 < 1.0

### 实现代码
```go
func calculateVolatilityRatio(boxKlines []models.Kline) float64 {
    if len(boxKlines) < 2 {
        return 0
    }

    // 计算收盘价的累计波动幅度
    totalMovement := 0.0
    for i := 1; i < len(boxKlines); i++ {
        diff := math.Abs(boxKlines[i].ClosePrice - boxKlines[i-1].ClosePrice)
        totalMovement += diff
    }

    // 计算箱体宽度
    boxWidth := boxKlines[0].HighPrice - boxKlines[0].LowPrice
    for _, k := range boxKlines {
        if k.HighPrice-k.LowPrice > boxWidth {
            boxWidth = k.HighPrice - k.LowPrice
        }
    }

    if boxWidth <= 0 {
        return 0
    }

    return totalMovement / boxWidth
}
```

在 `isValidBox` 中添加检查：
```go
volatilityRatio := calculateVolatilityRatio(boxKlines)
if volatilityRatio >= 0.85 { // 比率接近1或更高，视为单边行情
    return false
}
```

### 验证结果
- CYSUSDT 1h 周期：单边行情被正确过滤，无箱体
- CYSUSDT 15m 周期：正确识别出 2 个有效箱体

## 待优化项

1. **箱体延续逻辑的边界判断可以更精确**：当前使用收盘价判断，可以考虑使用最高价/最低价
2. **箱体类型可以细分**：range（矩形震荡）、ascend（上升通道）、descend（下降通道）
3. **信号强度计算可以更精细**：考虑箱体的震荡次数、价格收敛程度等因素

## 第六轮修复：回测中箱体时间顺序错误

### 问题
回测结果显示箱体的 `start_time` 晚于 `end_time`，例如：
- `start_time: "2026-03-26T03:45:00Z"`
- `end_time: "2026-03-25T18:30:00Z"`

### 根本原因
1. K线数据从数据库获取后是**从新到旧排列**的（`klines[0]` = 最新）
2. `runBacktestLoop` 中的循环遍历方式不正确，导致 `latestKline` 获取的是窗口中最旧的K线
3. `buildBoxFromSwingRange` 中的索引处理逻辑复杂且容易出错

### 修复方案

#### 1. 反转数据顺序
在 `runBacktestLoop` 中先将数据反转成从旧到新排列：
```go
// 反转K线数组：从旧到新排列
reversedKlines := make([]models.Kline, len(klines))
for i := 0; i < len(klines); i++ {
    reversedKlines[i] = klines[len(klines)-1-i]
}
```

#### 2. 修改循环遍历方向
从窗口起始位置开始，向后遍历到数据末尾：
```go
for i := 0; i <= len(reversedKlines)-windowSize; i++ {
    currentKline := reversedKlines[i+windowSize-1] // 窗口中最新的K线
    analysisWindow := reversedKlines[i : i+windowSize]
    // ...
}
```

#### 3. 简化 buildBoxFromSwingRange 和 isValidBox
由于数据已从旧到新排列，索引处理逻辑简化：
```go
// 直接使用正常索引范围
firstIdx := rangeSwings[0].Index
lastIdx := rangeSwings[len(rangeSwings)-1].Index
boxKlines := klines[firstIdx : lastIdx+1]
```

### 验证结果
修复前：
- `start_time: "2026-03-26T03:45:00Z"`
- `end_time: "2026-03-25T18:30:00Z"` ❌ (start > end)

修复后：
- `start_time: "2026-03-25T05:45:00Z"`
- `end_time: "2026-03-27T04:15:00Z"` ✅ (start < end)

## 第七轮修复：箱体 EndTime 被错误覆盖

### 问题
虽然时间顺序正确了，但箱体的 EndTime 被错误地设置为未来时间：
- 箱体检测时 EndTime=2026-03-25 07:15
- 但最终返回的 EndTime=2026-03-27 04:15

### 根本原因
1. 箱体被添加到 activeBoxes 后，立即被 checkBreakout 判定为突破
2. Analyze 方法中在箱体突破时会将 EndTime 设置为 latestKline.OpenTime，覆盖了原始值
3. 新K线到来时没有检查箱体是否已经被突破，导致无效延续

### 修复方案

#### 1. 添加检测时突破检查
在添加新箱体到 activeBoxes 之前，检查最新价格是否已经突破箱体边界：
```go
// 关键修复：检查箱体是否已经被突破
buffer := box.WidthPrice * 0.001
if latestPrice < box.LowPrice-buffer || latestPrice > box.HighPrice+buffer {
    // 箱体在检测时就已经被突破
    box.Status = models.BoxStatusClosed
    box.BreakoutPrice = &latestPrice
    a.boxes = append(a.boxes, box)
    continue // 不添加到 activeBoxes
}
```

#### 2. 保护 EndTime 不被覆盖
移除 checkBreakout 中覆盖 EndTime 的代码：
```go
// 箱体被突破后关闭
box.Status = models.BoxStatusClosed
// 注意：EndTime 保持为箱体检测时计算的值，不要覆盖
box.BreakoutPrice = &latestPrice
```

### 验证结果
修复后：
- `start_time: "2026-03-25T05:45:00Z"`
- `end_time: "2026-03-25T07:15:00Z"` ✅
- `klines_count: 7`
- 所有 7 根 K 线都在箱体边界内 ✅
- `breakout_price: 0.5611` ✅ (07:45收盘价，向下突破)
- `breakout_direction: "down"` ✅

## 补充修复 (2026-03-29)

### 问题：固定 width_threshold 不适应不同品种

`config.yml` 中 `width_threshold: 2.0` 是固定值，对 BTC（低波动）过宽，对山寨币（高波动）过窄，导致箱体要么全部过滤、要么误判。

### 修复方案：引入 ATR 动态阈值

#### 新增配置参数 (`config/config.go`, `config/config.yml`)
```yaml
strategies:
  box:
    use_dynamic_threshold: true
    atr_period: 14            # ATR 计算周期
    atr_multiplier: 2.0       # 阈值 = ATR均值/价格*100 * 倍数
    min_width_threshold: 0.3  # 最小宽度下限（%）
    max_width_threshold: 25.0 # 最大宽度上限（%）
    swing_lookback: 2         # 波峰波谷检测回溯K线数
```

#### calculateDynamicThreshold 方法
```go
func (s *BoxStrategy) calculateDynamicThreshold(klines []models.Kline) float64 {
    // 计算最近 atr_period 根K线的平均 ATR
    // threshold = (avgATR / latestClose * 100) * atrMultiplier
    // 限制在 [minWidthThreshold, maxWidthThreshold] 范围内
}
```

动态阈值效果示例：
- BTCUSDT 1h：ATR ~1500，价格 ~84000，阈值 ≈ 1.1%~1.6%
- 山寨币 15m：ATR/价格比更高，阈值自动调大

#### 修复 calculateVolatilityRatio 计算逻辑

原版用**累计路程（每根K线收盘价差绝对值之和）/ 单根K线宽度**，结果往往 >> 1.0，导致所有箱体都被拒绝。

修复为：**净位移（首尾收盘价差绝对值）/ 箱体极值宽度**。
- 震荡行情：价格来回，净位移远小于箱体宽度，比值小
- 趋势行情：价格单边移动，净位移接近箱体宽度，比值 >= 0.8 → 拒绝

### 回测验证结果 (2026-03-29)

BTCUSDT 1h, 2026-03-20 ~ 2026-03-28：
- 成功识别多个箱体，最大宽度 6.78%（71984→67412），跨越 114 根K线
- 最小箱体 1.72%（71731→70515），跨越 18 根K线

## 补充修复 (2026-03-29 下午)

### 问题4：K 线图表出现跳空

前端渲染 K 线图时，回测详情页面出现不应有的跳空（Bybit 行情不存在跳空）。

**根因**：`handleBacktestData` 中在更新完 K 线数据后，无论是信号视图还是箱体视图，都调用了 `scrollToTime(signalTime.value)`。当以箱体方式查看（非信号），`signalTime` 可能是箱体区间中点，导致图表跳转到错误位置，或触发 lightweight-charts 在不连续时间段渲染出视觉跳空。

**修复**：移除 `handleBacktestData` 末尾重复的 `scrollToTime` 调用，保留 `updateKlineData` 内已有的滚动逻辑。

另：`kline_repo.go` 中 `GetBySymbolPeriod` 的结束时间过滤使用 `close_time <= endTime`，会把跨越 endTime 的K线排除。改为 `open_time <= endTime`，确保时间范围内的所有K线都被返回，消除边界缺口。

### 问题5：箱体宽度过大，突破信号价值低

回测发现部分箱体宽度高达 6.78%、9.59%，跨越 100+ 根K线。这种宽箱体的突破信号已错过最佳进场时机。

**修复方案**：

1. **调整去重策略**：重叠度 > 70% 时，改为保留 `KlinesCount` 较少（更窄）的箱体，而非之前的 `KlinesCount` 较多（更宽）的箱体。适用于 `backtest_service.go` 和 `box_strategy.go` 两处。

2. **添加最大宽度过滤**：在 `filterValidBoxes`（backtest）和 `filterBoxes`（生产）中添加 `WidthPercent <= maxWidthThreshold` 过滤条件。

3. **降低 max_width_threshold**：`config.yml` 中从 `25.0` 降为 `5.0`，适配 BTC 1h 的合理箱体宽度范围。

4. **统一参数来源**：`newBoxStrategyAnalyzer` 改为从委托策略的 `Config()` 读取 `BoxStrategyConfig`，与 `config.yml` 保持一致，避免硬编码默认值与配置文件脱节。
